# 요구사항 문서

## 소개

Flow(flow.team) 스타일의 프로젝트 관리 SaaS 서비스의 핵심 기능을 정의한다. 칸반 보드 기반 업무 상태 관리, 업무 이관 워크플로우, 처리 결과 기록, 알림 시스템, 멀티 뷰 지원을 포함한다. AI 기능은 제외한다.

## 용어집

- **Board**: 프로젝트 내 업무 카드를 시각적으로 관리하는 칸반 보드
- **Column**: 보드 내 업무 상태를 나타내는 세로 단계 (예: 할 일, 진행 중, 완료)
- **Card**: 하나의 업무 단위를 나타내는 객체 (제목, 설명, 담당자, 마감일 등 포함)
- **Transfer**: 업무 이관. 현재 담당자가 다음 처리자에게 업무를 넘기는 행위
- **Resolution**: 처리 결과. 업무 완료/이관 시 기록하는 결과 유형 (승인/반려/완료/보류)
- **WIP_Limit**: Work In Progress 제한. 컬럼당 허용되는 최대 카드 수
- **Workflow_Chain**: 자동 이관 시 미리 정의된 처리자 순서
- **Project**: 업무 카드와 보드를 포함하는 최상위 관리 단위
- **Member**: 프로젝트에 참여하는 사용자
- **Subtask**: 카드 하위에 트리 구조로 연결된 하위 업무
- **Custom_Field**: 사용자가 정의하는 추가 필드
- **Notification_System**: 인앱, 이메일, 푸시 채널을 통해 알림을 전달하는 시스템
- **Dashboard**: 프로젝트 진행률, 병목 분석, 개인별 처리량을 시각화하는 화면
- **My_Tasks_View**: 현재 사용자에게 할당/이관된 업무 목록 화면

## 요구사항

### 요구사항 1: 프로젝트 기본 관리

**사용자 스토리:** 팀 리더로서, 프로젝트를 생성하고 멤버와 권한을 관리하고 싶다. 이를 통해 팀원들이 체계적으로 협업할 수 있다.

#### 인수 조건

1. Board는 사용자가 프로젝트를 이름, 설명, 공개 설정과 함께 생성, 조회, 수정, 삭제할 수 있어야 한다
2. 프로젝트가 생성되면, Board는 생성자를 전체 관리 권한을 가진 프로젝트 소유자로 지정해야 한다
3. Board는 최소 세 단계(소유자, 관리자, 멤버)의 멤버 역할 할당을 지원해야 한다
4. 멤버가 프로젝트에 추가되면, Board는 할당된 역할에 따라 권한을 부여해야 한다
5. 소유자 역할의 멤버가 제거될 때, Board는 제거 전에 다른 멤버에게 소유권 이전을 요구해야 한다

### 요구사항 2: 업무 카드 관리

**사용자 스토리:** 팀원으로서, 업무 카드를 생성하고 상세 정보를 관리하고 싶다. 이를 통해 업무 내용과 진행 상황을 명확히 파악할 수 있다.

#### 인수 조건

1. Board는 제목, 설명, 우선순위, 태그, 담당자, 시작일, 마감일을 포함한 Card 생성을 지원해야 한다
2. Board는 각 Card에 긴급, 높음, 보통, 낮음의 우선순위 수준을 지원해야 한다
3. Card가 생성되면, Board는 해당 Card를 보드의 지정된 Column에 배치해야 한다
4. Board는 무제한 깊이의 트리 구조로 Subtask 생성을 지원해야 한다
5. 상위 Card의 상태가 완료로 변경되면, Board는 모든 Subtask도 완료 상태인지 검증해야 한다
6. Board는 Card에 파일 첨부를 지원하며 업로드 및 다운로드 기능을 제공해야 한다
7. Board는 프로젝트별 Custom_Field 정의를 지원하며, 필드 유형은 텍스트, 숫자, 날짜, 단일선택, 다중선택이어야 한다
8. Board는 Card에 @ 표기를 사용한 멘션 기능이 포함된 댓글을 지원해야 한다
9. 댓글에서 사용자가 멘션되면, Notification_System은 멘션된 사용자에게 멘션 알림을 전송해야 한다

### 요구사항 3: 칸반 보드 및 컬럼 관리

**사용자 스토리:** 프로젝트 매니저로서, 칸반 보드의 컬럼을 자유롭게 설정하고 WIP 제한을 걸고 싶다. 이를 통해 팀의 업무 흐름을 최적화할 수 있다.

#### 인수 조건

1. Board는 보드 내 Column의 생성, 이름 변경, 순서 변경, 삭제를 지원해야 한다
2. Board는 각 Column에 양의 정수 또는 무제한으로 WIP_Limit 설정을 지원해야 한다
3. Card가 WIP_Limit에 도달한 Column으로 이동될 때, Board는 이동을 차단하고 경고 메시지를 표시해야 한다
4. Board는 Column 간 Card의 드래그 앤 드롭을 지원하여 Card 상태를 변경할 수 있어야 한다
5. Card가 드래그 앤 드롭으로 Column 간 이동되면, Board는 Card 상태를 대상 Column에 맞게 갱신해야 한다
6. Card가 Column 간 이동될 때, Board는 현재 담당자를 변경 없이 유지해야 한다
7. Card가 다른 Column으로 이동될 때, Board는 담당자를 동시에 변경할 수 있는 옵션을 제공해야 한다
8. Board는 Card 위치 변경을 WebSocket을 통해 1초 이내에 모든 연결된 사용자에게 반영해야 한다

### 요구사항 4: 멀티 뷰 지원

**사용자 스토리:** 팀원으로서, 동일한 업무 데이터를 보드, 리스트, 간트차트, 캘린더 등 다양한 뷰로 보고 싶다. 이를 통해 상황에 맞는 시각화로 업무를 파악할 수 있다.

#### 인수 조건

1. Board는 보드 뷰, 리스트 뷰, 간트차트 뷰, 캘린더 뷰의 네 가지 뷰 모드를 제공해야 한다
2. 사용자가 뷰를 전환할 때, Board는 동일한 Card 데이터를 선택된 뷰 형식으로 데이터 손실 없이 표시해야 한다
3. Board는 간트차트 뷰에서 Card를 시작일과 마감일 기준으로 타임라인에 배치하여 렌더링해야 한다
4. Board는 캘린더 뷰에서 Card를 마감일 기준으로 배치하여 렌더링해야 한다
5. Board는 리스트 뷰에서 Card를 Column 상태별로 그룹화하고, 우선순위, 마감일, 담당자 기준으로 정렬 가능하게 렌더링해야 한다
6. 어떤 뷰에서든 Card가 수정되면, Board는 해당 변경을 모든 다른 뷰에 실시간으로 반영해야 한다

### 요구사항 5: 업무 이관 (수동)

**사용자 스토리:** 담당자로서, 내 업무를 완료한 후 다음 처리자에게 이관하고 싶다. 이를 통해 워크플로우가 끊기지 않고 진행된다.

#### 인수 조건

1. 사용자가 수동 Transfer를 시작할 때, Board는 프로젝트 멤버 중에서 다음 담당자 선택을 요구해야 한다
2. Transfer가 시작될 때, Board는 승인, 반려, 완료, 보류 중 Resolution 유형 선택을 요구해야 한다
3. Transfer가 시작될 때, Board는 이관 코멘트 입력과 파일 첨부를 허용해야 한다
4. 반려 유형의 Resolution이 포함된 Transfer가 기록되면, Board는 해당 Card를 이전 담당자에게 자동으로 재할당해야 한다
5. Board는 각 Transfer 이벤트를 타임스탬프, 발신자, 수신자, Resolution, 코멘트와 함께 타임라인에 기록해야 한다
6. Transfer가 완료되면, Notification_System은 새 담당자에게 이관 수신 알림을 전송해야 한다

### 요구사항 6: 업무 이관 (자동 워크플로우)

**사용자 스토리:** 프로젝트 매니저로서, 워크플로우 체인을 미리 정의하여 업무가 자동으로 다음 처리자에게 이관되게 하고 싶다. 이를 통해 반복적인 이관 작업을 자동화할 수 있다.

#### 인수 조건

1. Board는 Column 전환별로 순서가 지정된 담당자 목록으로 Workflow_Chain 정의를 지원해야 한다
2. Card가 Workflow_Chain이 정의된 Column에 도달하고 현재 단계가 완료되면, Board는 체인의 다음 담당자에게 자동으로 Transfer를 생성해야 한다
3. 자동 Transfer가 발생할 때, Board는 Resolution 기록을 포함하여 수동 Transfer와 동일한 규칙을 적용해야 한다
4. Workflow_Chain의 마지막 단계가 완료되면, Board는 해당 Card를 완료 상태로 표시해야 한다
5. Board는 이미 진행 중인 Card에 영향을 주지 않고 Workflow_Chain 수정을 허용해야 한다

### 요구사항 7: 처리 결과 기록

**사용자 스토리:** 담당자로서, 업무 완료 시 처리 결과를 기록하고 싶다. 이를 통해 다음 처리자가 이전 작업 결과를 확인할 수 있다.

#### 인수 조건

1. Board는 승인, 반려, 완료, 보류 유형의 Resolution 기록을 지원해야 한다
2. 프로젝트 설정에서 Resolution 필수가 활성화된 경우, Board는 Card를 완료 처리하거나 이관하기 전에 Resolution 입력을 강제해야 한다
3. Resolution이 기록될 때, Board는 결과 코멘트와 파일 첨부를 허용해야 한다
4. Transfer가 발생할 때, Board는 다음 담당자에게 전송되는 이관 알림에 Resolution 상세 내용을 포함해야 한다
5. Board는 Card 상세 화면에서 전체 Resolution 이력을 시간순으로 표시해야 한다

### 요구사항 8: 알림 시스템

**사용자 스토리:** 팀원으로서, 업무 이관, 처리 결과, 마감 임박 등의 알림을 받고 싶다. 이를 통해 중요한 업무 변경을 놓치지 않을 수 있다.

#### 인수 조건

1. Transfer를 수신하면, Notification_System은 새 담당자에게 "내 차례입니다" 메시지와 함께 알림을 전송해야 한다
2. 이관된 Card에 Resolution이 기록되면, Notification_System은 다음 담당자에게 "이전 단계 완료" 메시지와 함께 알림을 전송해야 한다
3. 반려 유형의 Resolution이 포함된 Transfer가 발생하면, Notification_System은 이전 담당자에게 "재작업 필요" 메시지와 함께 알림을 전송해야 한다
4. Card 마감일이 3일 전이면, Notification_System은 Card 담당자에게 D-3 마감 임박 알림을 전송해야 한다
5. Card 마감일이 1일 전이면, Notification_System은 Card 담당자에게 D-1 마감 임박 알림을 전송해야 한다
6. Card 마감일이 당일이면, Notification_System은 Card 담당자에게 D-day 알림을 전송해야 한다
7. Card 마감일이 지났으면, Notification_System은 Card 담당자와 프로젝트 관리자에게 기한 초과 알림을 전송해야 한다
8. Notification_System은 인앱, 이메일, 푸시 알림의 세 가지 채널을 통해 알림을 전달해야 한다
9. Notification_System은 각 멤버가 채널별, 알림 유형별로 알림 수신 설정을 구성할 수 있도록 해야 한다

### 요구사항 9: 칸반과 이관의 관계

**사용자 스토리:** 팀원으로서, 칸반 이동과 업무 이관이 독립적으로 또는 동시에 동작하길 원한다. 이를 통해 상태 변경과 담당자 변경을 유연하게 처리할 수 있다.

#### 인수 조건

1. Card가 Transfer 없이 Column 간 이동될 때, Board는 Card 상태만 변경하고 현재 담당자를 유지해야 한다
2. Column 변경 없이 Transfer가 시작될 때, Board는 담당자만 변경하고 현재 Column 위치를 유지해야 한다
3. Card가 다른 Column으로 이동될 때, Board는 사용자에게 "담당자도 변경하시겠습니까?" 옵션을 표시해야 한다
4. Column 이동 시 사용자가 담당자 변경을 확인하면, Board는 Column 이동과 Transfer를 하나의 원자적 작업으로 실행해야 한다
5. Column 이동과 Transfer가 동시에 발생하면, Board는 두 이벤트를 모두 Card 타임라인에 기록해야 한다

### 요구사항 10: 내 업무 뷰

**사용자 스토리:** 팀원으로서, 나에게 할당되거나 이관된 업무를 한눈에 보고 싶다. 이를 통해 내가 처리해야 할 업무를 빠르게 파악할 수 있다.

#### 인수 조건

1. My_Tasks_View는 모든 프로젝트에 걸쳐 현재 사용자에게 할당되거나 이관된 모든 Card를 표시해야 한다
2. My_Tasks_View는 진행 중, 완료, 기한 초과 상태별 필터링을 지원해야 한다
3. My_Tasks_View에서 Transfer를 통해 수신된 Card의 경우, 발신자의 이전 Resolution과 이관 코멘트를 표시해야 한다
4. My_Tasks_View는 마감일 기준으로 Card를 정렬하되, 기한 초과 Card를 최상단에 표시해야 한다
5. My_Tasks_View는 이관된 Card에 대해 이관 출처(발신자 이름 및 타임스탬프)를 표시해야 한다

### 요구사항 11: 대시보드

**사용자 스토리:** 프로젝트 매니저로서, 프로젝트 진행률과 병목 지점을 시각적으로 파악하고 싶다. 이를 통해 프로젝트를 효과적으로 관리할 수 있다.

#### 인수 조건

1. Dashboard는 완료된 Card 수 대비 전체 Card 수를 기반으로 프로젝트 전체 진행률을 백분율로 표시해야 한다
2. Dashboard는 병목 단계를 식별하기 위해 Column별 평균 체류 시간을 표시해야 한다
3. Dashboard는 기간별 완료 Card 수로 개인별 멤버 처리량을 표시해야 한다
4. Dashboard는 일간, 주간, 월간 단위의 기간 선택을 지원해야 한다
5. Card가 Column 간 이동하면, Dashboard는 5초 이내에 체류 시간 지표를 갱신해야 한다
6. Dashboard는 Column별 Card 수를 보여주는 Column 분포 시각 차트를 표시해야 한다

### 요구사항 12: 실시간 동기화

**사용자 스토리:** 팀원으로서, 다른 팀원의 변경사항을 실시간으로 확인하고 싶다. 이를 통해 항상 최신 상태의 보드를 볼 수 있다.

#### 인수 조건

1. Board는 각 활성 사용자 세션에 대해 WebSocket 연결을 수립해야 한다
2. Card 속성이 수정되면, Board는 동일 프로젝트를 보고 있는 모든 연결된 사용자에게 1초 이내에 변경 사항을 브로드캐스트해야 한다
3. Card가 드래그 앤 드롭으로 이동되면, Board는 모든 연결된 사용자에게 실시간으로 위치 변경을 브로드캐스트해야 한다
4. WebSocket 연결이 끊어지면, Board는 지수 백오프 방식으로 자동 재연결을 시도하고, 재연결 시 누락된 변경 사항을 동기화해야 한다
5. 여러 사용자가 동시에 동일한 Card를 수정하려 할 때, Board는 낙관적 잠금을 적용하고 나중에 수정한 사용자에게 충돌을 알려야 한다
